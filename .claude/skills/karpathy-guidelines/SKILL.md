# Karpathy Guidelines — 백엔드 각색

Andrej Karpathy가 지적한 LLM 코딩 실패 패턴(모델이 틀린 가정을 하고, 코드를 과하게 만들고, 무관한 부분을 건드림)을 줄이는 4원칙. 원전: `forrestchang/andrej-karpathy-skills`(MIT). 여기서는 Skein 백엔드(NestJS·TypeORM·Supabase·멀티테넌시) 맥락으로 각색했다.

> **트레이드오프(원전 명시)**: 이 원칙들은 속도보다 신중을 택한다. 사소한 작업엔 판단껏. 단, **테넌시·인증·과금·마이그레이션은 사소하지 않다** — 항상 전수 적용.

관련: 가정 검증은 [[devils-advocate]], 검증 사실은 `.claude/harness/backend-reference.md`, 컨텍스트 원칙은 `.claude/harness/principles.md`.

---

## 1. Think Before Coding — 코딩 전에 생각

**가정하지 마라. 헷갈림을 숨기지 마라. 트레이드오프를 드러내라.**

구현 전:
- 가정을 명시하라. 불확실하면 물어라.
- 해석이 여러 개면 전부 제시하라 — 조용히 하나 고르지 마라.
- 더 단순한 방법이 있으면 말하라. 근거 있으면 반박하라.
- 불명확하면 멈춰라. 무엇이 헷갈리는지 이름 붙이고 물어라.

**백엔드 적용**: "이 엔드포인트는 어느 워크스페이스 스코프인가?", "테넌트 경계는 workspace인가 brand인가?", "이 필드는 클라이언트 입력인가 서버 결정값인가?"를 **코드 쓰기 전에** 확정. 워크스페이스 스코핑에 대한 잘못된 가정은 **보안 버그(BOLA/IDOR)**다.

## 2. Simplicity First — 단순함 우선

**문제를 푸는 최소 코드. 투기적 코드 금지.**

- 요청되지 않은 기능 금지.
- 일회용 코드에 추상화 금지.
- 요청 안 된 "유연성"·"설정 가능성" 금지.
- 불가능한 시나리오용 에러 처리 금지.
- 200줄 썼는데 50줄로 되면 다시 써라.

스스로 물어라: "시니어 엔지니어가 과설계라고 할까?" 그렇다면 단순화.

**백엔드 적용**: 컨트롤러는 얇게(검증·라우팅), 로직은 서비스. 추측성 추상 레이어·제네릭 베이스 클래스·미사용 DI 프로바이더 금지. CLAUDE.md 모듈 규약을 넘는 구조 도입 금지.

## 3. Surgical Changes — 외과적 변경

**꼭 필요한 것만 건드려라. 네가 만든 잔해만 치워라.**

기존 코드 편집 시:
- 인접 코드·주석·포맷을 "개선"하지 마라.
- 안 깨진 것 리팩터하지 마라.
- 기존 스타일을 따르라(네 취향과 달라도).
- 무관한 데드 코드를 발견하면 — 언급만 하고 지우지 마라.

네 변경이 고아를 만들면:
- 네 변경이 미사용으로 만든 import/변수/함수만 제거.
- 기존 데드 코드는 요청 없이 제거 금지.

테스트: 변경된 모든 줄이 사용자 요청으로 직접 추적되어야 한다.

**백엔드 적용**: 마이그레이션·엔티티·DTO는 미러링 관계 — 한쪽만 손대고 다른 쪽 방치 금지. `synchronize: false` 등 불변 설정 절대 변경 금지. 깨끗한 diff가 리뷰 통과의 전제(CLAUDE.md).

## 4. Goal-Driven Execution — 목표 주도 실행

**성공 기준을 정의하라. 검증될 때까지 반복하라.**

작업을 검증 가능한 목표로 변환:
- "검증 추가" → "잘못된 입력에 대한 테스트를 쓰고, 통과시켜라"
- "버그 수정" → "재현하는 테스트를 쓰고, 통과시켜라"
- "X 리팩터" → "전후로 테스트가 통과하는지 확인하라"

다단계 작업은 간단한 계획을 명시:
```
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
```

**백엔드 적용 — 표준 검증 루프**:
- 모든 코드 변경 후 **`npm run build`로 타입 검증**(CLAUDE.md 필수).
- 새 로직엔 테스트 추가(`npm run test`, e2e `npm run test:e2e`) — 가능하면 실패 테스트 먼저.
- 테넌시 회귀 테스트: 멤버 아닌 유저의 접근이 거부되는지(`assertWorkspaceMember`).
- 강한 성공 기준이 독립 반복을 가능케 한다. 약한 기준("작동하게")은 매번 재확인을 부른다.

## 효과 측정 (자가 점검)
이 원칙이 작동 중이라면: diff에 불필요한 변경이 줄고, 재작업이 줄고, 테넌시/과금 가정 오류가 줄고, "이건 왜 바꿨어?"가 줄어든다.

## Anti-Triggers
- 단순 팩트 조회·문법 질문·번역
- 이미 명확한 단순 작업의 즉시 실행
- 순수 리서치 → `/osint-deepsearch`, 설계 비평 → `/devils-advocate`
