# 기능명세서 예시 — 범용 "목록 조회 화면"

서비스 공통 패턴인 "목록 조회" 화면을 기반으로 기능명세서 작성 Before/After를 비교한다.
실제 프로젝트에서 이 패턴을 가장 많이 재사용하므로 품질 기준으로 삼는다.

---

## Before — 기존 방식 (Figma 텍스트·불릿 나열)

Figma 프레임 안에 텍스트 레이어로 삽입된 방식. 흔히 발견되는 문제 패턴.

```
목록 보기 화면

- 로그인한 사용자가 접근 가능
- 목록을 카드 형식으로 보여줌
- 최대 10개씩 표시
- 더보기 버튼으로 추가 로드
- 빈 상태일 때 안내 문구 표시
- 모바일에서는 다르게 보임
```

**문제점:**
- 기능 ID 없음 → 화면설계서·IA에서 교차 참조 불가
- 입력/처리/출력 구분 없음 → 개발자가 API 스펙 파악 불가
- 엣지케이스(에러·권한 없음·로딩) 미정의 → QA 케이스 누락
- 브레이크포인트 "다르게 보임" 수준 → 구체적 레이아웃 차이 불명확
- Figma 내 텍스트라 검색·링크·버전 관리 불가

---

## After — 개선된 표 형식 (Notion 저장 + Figma 링크)

### 문서 헤더

```yaml
---
client: [클라이언트ID]
project: [서비스명_기능명세]
type: functional-spec
created: YYYY-MM-DD
version: v1.0
prd_version: [기준 PRD 버전]
approved: false
---
```

| 항목 | 내용 |
|------|------|
| 서비스명 | [서비스명] |
| 기준 PRD | prd-[서비스].md ([버전]) |
| 브레이크포인트 | Mobile ≤767px / PC ≥768px |
| 작성일 | YYYY-MM-DD |
| 버전 | v1.0 |

---

### 기능 ID 테이블 (목록 화면)

| 기능ID | 기능명 | 화면명 | 우선순위 | PRD 참조 |
|--------|--------|--------|----------|----------|
| F001 | 목록 조회 | 목록 화면 | Must-Have | §3.1 |
| F002 | 더보기 로드 | 목록 화면 | Must-Have | §3.1 |
| F003 | 빈 상태 노출 | 목록 화면 | Must-Have | §3.1 |
| F004 | 항목 클릭 → 상세 이동 | 목록 화면 | Must-Have | §3.2 |

---

### 목록 화면 (F001~F004)

#### 1. 표출 정보

| # | 요소명 | 설명 |
|---|--------|------|
| 1 | 헤더/GNB | 서비스 로고 + 사용자 상태 (로그인 여부) |
| 2 | 페이지 타이틀 | "[서비스명] 목록" |
| 3 | 목록 카드 | 항목명, 생성일, 상태 뱃지 |
| 4 | 더보기 버튼 | 추가 10개 로드 (로딩 아이콘 포함) |
| 5 | 빈 상태 영역 | 항목 0개일 때 대체 노출 |

#### 2. 인터랙션 명세 (7컬럼 표)

| 기능ID | 화면명 | 트리거 | 입력값 | 처리 | 출력 | 엣지케이스 |
|--------|--------|--------|--------|------|------|------------|
| F001 | 목록 화면 | 페이지 진입 (로그인) | user_id (세션) | `GET /api/items?limit=10&sort=latest&user={id}` | 카드 그리드 렌더링 (최신순 10개) | 데이터 없음→F003 / API 실패→에러 메시지 + [다시 시도] / 비로그인→/login 리다이렉트 |
| F002 | 목록 화면 | [더보기] 버튼 클릭 | page_cursor (마지막 항목 ID) | `GET /api/items?limit=10&cursor={last_id}` 추가 호출 | 기존 목록 아래 10개 추가 렌더링 | 마지막 페이지: 버튼 숨김 / API 실패: 버튼 비활성 + "불러오기 실패" 토스트 |
| F003 | 목록 화면 | 항목 0개 조회 결과 수신 | — | 빈 상태 UI 조건 판별 | "[서비스명]이 없습니다" + [새로 만들기] CTA | — |
| F004 | 목록 화면 | 항목 카드 클릭 | item_id | 클라이언트 사이드 라우팅 | /items/{id} 페이지로 이동 | 존재하지 않는 ID: 404 / 권한 없음: "접근 권한 없음" 토스트 |

#### 3. 상태 정의

| 상태 | 조건 | UI 처리 |
|------|------|---------|
| 정상 | 항목 1개 이상 조회 성공 | 카드 그리드 렌더링 |
| 로딩 중 | 최초 진입 시 API 응답 대기 | 스켈레톤 UI — 카드 형태 회색 블록 3개 |
| 더보기 로딩 | [더보기] 클릭 후 API 응답 대기 | 버튼 내 로딩 아이콘 360도 무한 회전 |
| 빈 상태 | 항목 0개 | "[서비스명]이 없습니다" + [새로 만들기] 버튼 |
| 에러 | API 실패 / 타임아웃 | "데이터를 불러오지 못했습니다" + [다시 시도] 버튼 |
| 비로그인 | 세션 없음 | /login 리다이렉트, 로그인 성공 후 원래 경로 복귀 |

#### 4. 브레이크포인트 분기

| 구분 | Mobile (≤767px) | PC (≥768px) |
|------|-----------------|-------------|
| 카드 레이아웃 | 1열 풀너비 리스트 | 3열 그리드 (gap 24px) |
| 네비게이션 | 하단 탭바 고정 | 좌측 사이드바 (240px) |
| 더보기 버튼 | 전체 너비 (full-width) | 우측 정렬, 너비 120px |
| 빈 상태 CTA | 하단 고정 버튼 | 빈 상태 영역 중앙 버튼 |

## Before/After 비교 요약

| 항목 | Before (Figma 텍스트·불릿) | After (7컬럼 표·Notion 저장) |
|------|---------------------------|------------------------------|
| 기능 식별 | 없음 (기능명만) | F001~F004 ID로 교차 참조 가능 |
| 처리 로직 | "목록을 보여줌" (모호) | API 엔드포인트 + 파라미터 명시 |
| 엣지케이스 | "빈 상태일 때 안내 문구" (단일) | 빈 상태·에러·권한 없음·로딩 4종 정의 |
| 브레이크포인트 | "모바일에서는 다르게 보임" | 레이아웃·컴포넌트·버튼 너비 구체 명시 |
| 저장 위치 | Figma 텍스트 레이어 (검색 불가) | Notion 페이지 저장 + Figma 링크 참조 |
| 상태 정의 | 없음 | 정상·로딩·빈 상태·에러·비로그인 5종 |
| QA 활용성 | 낮음 (케이스 추론 필요) | 높음 (엣지케이스 표에서 직접 추출) |
