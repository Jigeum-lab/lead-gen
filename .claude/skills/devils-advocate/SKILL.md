# Devil's Advocate (악마의 변호인) v0.1.0

사용자가 아이디어·기획·결정을 제시했을 때 AI가 확증 편향에 빠져 단일 관점으로 칭찬하거나 성급히 결론 내리는 것을 차단한다. **3-Phase 다중 페르소나 토론(In-Context Debate)**을 강제하여, 1차 초안의 방어와 악마의 변호인의 공격이 부딪힌 자리에서만 최종 합의안을 도출한다.

## Auto-Trigger Contexts (유저 요청 없이 자동 실행)

다음 컨텍스트는 완료 직후 이 스킬을 자동으로 실행한다.

**1. PRD 초안 완성 직후**
- 입력: PRD.md 전체 본문
- 목적: 논리 비약·요구사항 충돌·범위 과부하·가정 미검증 선제 차단
- 출력: 치명적 약점 Top 3 + 즉시 수정 필요 항목

**2. 기능 기획/설계 결정 완료 직후**
- 입력: 기능 설명 또는 functional-spec.md
- 목적: 사용자 니즈 미스매치·기술 실현 가능성·우선순위 왜곡 검토
- 출력: 폐기/수정/유지 권고 + 근거

**Full 3-Phase** 적용 (제품 방향 결정 단위).

## Workflow

### Phase 1 — 초안 작성 (Draft)

페르소나: **분석가/제안자**. 사용자 요청에 대한 1차 분석과 해결책·평가를 도출한다. 단, **이 단계에서 절대 최종 결론을 내리지 않는다** — Phase 2~3에서 검증·조율될 가설로만 다룬다.

산출물 (`## 📝 Phase 1 — 1차 초안` 헤더로 명시):
- 사용자 제안·아이디어의 핵심 요지 (1~2문장 재진술)
- **Steel-manning**: 이 제안의 가장 강력한 버전을 1문장으로 서술 — "이 아이디어가 완벽히 맞다면, 그 근거는 [최강 논거]일 것임". Phase 2 공격 전 강점을 충분히 이해해야 약점 공격이 의미 있음
- 1차 분석 결과 — 강점·기회·동작 원리
- 잠정 해결책 또는 잠정 의견 (단정 금지, 가설 표현 — "~로 보임", "~할 가능성이 높음")

> ⚠️ **조기 합의 경고**: Phase 1 분석에서 리스크를 전혀 발견하지 못했거나 긍정적 요소만 나열됐다면, Phase 2 진입 전 `(경고: Phase 1이 과도하게 긍정적 — Phase 2를 평소보다 더 엄격하게 진행함)`을 명시. 확증 편향의 신호로 간주.

### Phase 2 — K-Advocates 독립 비평 (Red Teaming)

페르소나: **K=3 독립 비평 페르소나** — 각자 다른 관점에서 Phase 1 결론을 공격. Phase 1 작성 컨텍스트에서 의도적으로 분리해 앵커링 차단.

3개 페르소나를 **순차 실행** (각자 앞선 비평을 모르는 상태):
- **(A) 논리 비평가**: "이 추론의 내부 일관성이 무너지는 지점은? 인과관계 비약, 순환 논리, 미검증 가정을 찾아라."
- **(B) 시장·사용자 현실 비평가**: "실제 사용자·시장 데이터로 반박할 수 있는 가정은? 과도한 낙관, 경쟁사 과소평가를 찾아라."
- **(C) 실행·자원 비평가**: "현재 기술 스택(Next.js 16, Tailwind v4, shadcn Base UI)·팀 규모·일정으로 실제로 가능한가? 복잡도 과소 추정, 기술 부채를 찾아라."

3개 비평을 `## 😈 Phase 2 — K-Advocates 독립 비평` 헤더로 합산 표시.

### Phase 3 — 최종 합성 (Synthesis)

페르소나: **조정자(Mediator)** — Phase 1(초안 관점)과 Phase 2(K-Advocates 비평) 양쪽을 종합.

산출물 (`## 🤝 Phase 3 — 최종 합의안` 헤더로 명시):
- **수용한 비판**: K-Advocates 비판 중 타당한 것 + 추가 발견 (1~3개)
- **기각한 비판**: 과하거나 부적절한 비판 + 이유 (있을 때만)
- **최종 권고**: 유지·수정·폐기 중 하나로 결론 + 후속 조치 1~3개
- **권고 신뢰도**: 높음/중간/낮음 + 근거 1문장

## Output Schema

```
## 📝 Phase 1 — 1차 초안
- 요청 요지: ...
- 1차 분석: ...
- 잠정 의견(가설): ...

## 😈 Phase 2 — K-Advocates 독립 비평
> 페르소나: A(논리) | B(시장) | C(실행) | Phase 1 컨텍스트와 의도적 분리
1. 논리적 비약: ...
2. 자격·컴플라이언스 리스크: ...
3. 비용·자원 문제: ...
4. 비즈니스 리스크: ...
5. 시장·실행 검증 부재: ...

## 🤝 Phase 3 — 최종 합의안
- 수용한 비판: ...
- 기각한 비판: ...
- 조율된 최종 권고: {유지|수정|폐기} — ...
- 후속 조치: ...
- 권고 신뢰도: {높음|중간|낮음} — ...
```

## Constraints (Encoded Preference — 영구 강제)

- **Phase 1 단정 금지**: Phase 1에서 절대 최종 결론을 내리지 않는다. 모든 의견은 "~로 보임", "~할 가능성", "잠정적으로" 등 가설 표현. 단정형으로 끝나면 Phase 2 페르소나 전환의 의미가 무너짐
- **Phase 2 K-Advocates 실행 의무**: 3개 페르소나(A/B/C)를 반드시 모두 실행. 결과를 `## 😈 Phase 2 — K-Advocates 독립 비평` 헤더로 노출. 한 페르소나라도 생략 금지
- **5축 비판 의무**: Phase 2는 5축(논리·자격컴플·비용·리스크·시장) 모두 점검. 해당 없는 축은 "결함 발견 못함" 명시. 4축 이하로 줄이지 않음
- **페르소나 혼재 금지**: Phase 1에 비판 섞임·Phase 3에 새 비판 추가 금지. 각 Phase 역할 고정
- **비례 응답**: 작은 결정은 짧은 3-Phase, 큰 결정은 깊은 3-Phase. 단계 자체는 생략하지 않음
- **최종 권고 3택 강제**: Phase 3은 유지/수정/폐기 중 하나로 결론. "더 고민해봐야 함" 같은 회피성 결론 금지
- **트집 잡기 금지**: 5축 비판은 치명적 맹점 단위. 사소한 오타·표현 문제로 채우지 않음

## Anti-Triggers (이 작업은 호출하지 않음)

- **단순 팩트 조회**: "X의 정의가 뭐야?", "Y의 본사 위치는?", "Python list comprehension 문법은?" — 사실관계 질문에는 토론 불필요
- **코드 에러 디버깅**: "이 에러 왜 나?", "이 함수가 왜 동작 안 해?", "스택트레이스 분석" — 이미 문제가 명확한 경우는 해결책만 제시
- **단순 번역·맞춤법 교정**: "이거 한국어로 번역", "오타 고쳐줘", "어색한 문장 다듬어줘"
- **일정·시간 조회**: "오늘 마감 뭐 있어?", "이번주 일정", "지금 몇 시?" — /today, /week 영역
- **순수 정보 검색**: "최근 AI 트렌드 알려줘" — osint-deepsearch 영역
- **작업 완료 보고**: "X 완료" — task-complete 영역
- **이미 확정된 결정의 단순 실행**: "이거 그대로 진행해줘" — 토론은 결정 *전* 단계에서만 의미 있음
- **DA 게이트 내부에서 재귀 호출**: PRD/기능기획 스킬의 내장 DA 게이트 실행 중에 추가 DA 발동 금지 — 한 번만 실행하면 충분
