# QA Skill

PRD·와이어프레임·생성 코드(src/)를 3개 전문가 관점에서 병렬 검토. **관대한 판정 금지** — 기계적 체크리스트 기반으로만 출력.

## Phase 1 — Input Gate

### HARD-GATE 1: PRD 승인 확인
`product/{project}/PRD.md` 존재 여부 + frontmatter `approved: true` 확인.
→ 없거나 미승인: `❌ [HARD-GATE] PRD.md가 없거나 approved: true가 아닙니다. /prd를 먼저 완료하고 승인받으세요.`

### HARD-GATE 2: 와이어프레임 확인
`product/{project}/ux/wireframes.md` 존재 여부 확인.
→ 없음: `❌ [HARD-GATE] wireframes.md가 없습니다. /wireframe을 먼저 완료하세요.`

### SOFT-GATE: 코드 파일 확인
`src/` 디렉토리 존재 여부 확인.
→ 없음: `⚠️ [SOFT-GATE] src/ 코드가 없습니다. /hi-fi 없이 PRD+와이어프레임 기반 부분 QA를 진행하시겠습니까?`
  - AUQ: "부분 QA 진행 | /hi-fi 완료 후 재시작"

### Output Gate: qa-review.md 존재 시
`product/{project}/qa-review.md` 존재 확인.
→ 있음: 버전·생성일·Critical 건수 표시 후 AUQ: "기존 QA 보고서 열람 | 특정 영역 재검토 | 전체 재실행"

---

## Phase 2 — 컨텍스트 조립

1. `PRD.md` 로드 → P0 기능 목록 추출 (기능ID·기능명·우선순위)
2. `wireframes.md` 로드 → 화면 목록 추출 (화면ID·화면명·기능 참조)
3. `src/` 코드 있으면:
   - 화면별 `page.tsx` + `_components/` 파일 읽기
   - shadcn/Base UI API 오용 여부 (`asChild` 사용, hex 하드코딩) 목록화
4. 검토 컨텍스트 패킷 구성:
   ```
   [REVIEW_CONTEXT]
   - P0 기능 목록: N개
   - 화면 목록: N개
   - 코드 파일: 있음/없음
   ```

## Phase 3 — 병렬 QA 검토 (3 서브에이전트 동시 실행)

아래 3개를 **동시에** 실행. 각 서브에이전트는 qa-reviewer 역할.

### Sub-agent A: PRD 요건 충족 검토

**페르소나**: PRD 컴플라이언스 전문가. P0 기능 하나도 빠짐없이 추적.

각 P0 기능에 대해:
- 와이어프레임에 대응 화면이 존재하는가?
- Figma에 해당 인터랙션이 구현되어 있는가?
- 에러 상태·빈 상태·로딩 상태가 정의되어 있는가?
- KPI와 연결된 핵심 플로우가 커버되는가?

출력 형식:
```
| 기능ID | 기능명 | 우선순위 | 충족 여부 | 비고 |
|---|---|---|---|---|
| F001 | ... | P0 | ✅ / ❌ / ⚠️ | ... |
```

Critical 기준: P0 기능 누락 또는 핵심 플로우 단절.

### Sub-agent B: WCAG 2.1 AA 접근성 검토

**페르소나**: 디지털 접근성 전문가, WCAG 2.1 AA 기준 엄격 적용.

검토 항목:
- **색상 대비**: 일반 텍스트 4.5:1 이상 / 대형 텍스트(18pt+ 또는 14pt Bold+) 3:1 이상 / 비텍스트 UI 요소 3:1 이상
- **텍스트 크기**: 최소 14sp(Android) / 17pt(iOS) — 본문 기준
- **터치 타겟**: 최소 44×44px (모바일)
- **키보드 내비게이션**: 모든 인터랙션 요소 키보드 접근 가능
- **포커스 표시**: 모든 포커스 가능 요소에 visible focus state
- **스크린 리더**: 의미 있는 이미지 alt 텍스트 / 폼 input에 label 연결
- **오류 식별**: 에러 메시지 색상 외 텍스트로도 표시

출력 형식:
```
| 항목 | WCAG 기준 | 검토 결과 | 심각도 | 이슈 위치 |
|---|---|---|---|---|
| 텍스트 대비 | 1.4.3 (AA) | ✅ / ❌ | Critical/Major/Minor | 화면명/컴포넌트 |
```

Critical 기준: WCAG 2.1 AA 위반 (색상 대비 미달·터치 타겟 미달·포커스 없음).

### Sub-agent C: 닐슨 노만 10 휴리스틱 검토

**페르소나**: UX 리서처·휴리스틱 평가 전문가. 객관적 증거 기반 평가.

10개 휴리스틱별 평가 (1=심각 문제 / 5=우수):

| # | 휴리스틱 | 체크 포인트 |
|---|---|---|
| 1 | 시스템 상태 가시성 | 로딩·저장·완료·오류 상태 표시 |
| 2 | 실제 세계와의 일치 | 사용자 언어 사용, 기술 용어 없음 |
| 3 | 사용자 통제·자유 | 되돌리기(Undo), 나가기, 취소 경로 |
| 4 | 일관성·표준 | 동일 기능 동일 레이블·패턴 |
| 5 | 에러 예방 | 위험 액션 확인 다이얼로그, 입력 유효성 |
| 6 | 기억 부담 최소화 | 선택지 가시화, 컨텍스트 유지 |
| 7 | 유연성·효율성 | 숙련자 단축키, 자주 쓰는 기능 쉬운 접근 |
| 8 | 미적·최소주의 | 불필요 정보 없음, 정보 위계 명확 |
| 9 | 에러 복구 지원 | 에러 메시지 = 상황+이유+해결책 |
| 10 | 도움말·문서 | 복잡 기능에 컨텍스트 도움말 |

출력 형식:
```
| # | 휴리스틱 | 점수 (1-5) | 주요 발견 | 권고 사항 |
|---|---|---|---|---|
| 1 | 시스템 상태 가시성 | 3 | ... | ... |
```

심각도 분류: 점수 1-2 → Critical/Major / 3 → Minor / 4-5 → Info

## Phase 4 — 통합 + 우선순위화

오케스트레이터가 3개 보고서 수집 후:

1. **이슈 분류**:
   - `Critical`: 사용자 여정 차단 또는 WCAG AA 위반 → 즉시 수정 필수
   - `Major`: 사용성 심각 저하 → 다음 스프린트 내 수정
   - `Minor`: 개선 권고 → 백로그 등록
   - `Info`: 미래 고려 사항

2. **중복 제거**: 여러 에이전트가 같은 이슈를 발견한 경우 가장 심각한 분류로 통합

3. **수정 로드맵**: Critical → Major → Minor 순서로 정렬된 액션 리스트

## Constraints

- Fail 판정에는 반드시 **위치 + 개선 방법** 명시. 위치 없는 Fail 금지.
- "대체로 괜찮음" 판정 금지. Pass / Fail / N/A만.
- 코드 없이 진행 시 모든 항목에 `(코드 미확인)` 태그 부착.
- 자기평가 금지: 이 스킬이 만든 산출물(PRD, wireframe)을 이 스킬이 QA하지 않음.
- Critical 0건이어도 Minor·Info 항목 기술 필수.

## Anti-Triggers

- PRD 작성 → `/prd` 스킬
- 와이어프레임 작성 → `/wireframe` 스킬
- 코드 생성 → `/hi-fi` 스킬
- 기능명세서 → `/functional-spec` 스킬
