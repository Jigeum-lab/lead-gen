# 기능명세서 표 형식 기준

기능명세서 작성 시 준수해야 하는 표 구조·섹션 구성·엣지케이스 작성 기준을 정의한다.

---

## 1. 기본 표 구조 (7컬럼 최소)

```markdown
| 기능ID | 화면명 | 트리거 | 입력값 | 처리 | 출력 | 엣지케이스 |
|--------|--------|--------|--------|------|------|------------|
| F001   | 홈     | 페이지 진입 | — | 프로젝트 목록 10개 조회 (최신순) | 카드 그리드 렌더링 | 데이터 없음: "아직 프로젝트가 없습니다" + [새 프로젝트 만들기] CTA 버튼 |
```

### 컬럼 정의

| 컬럼 | 타입 | 설명 | 예시 |
|------|------|------|------|
| 기능ID | 필수 | F001~Fn 순번 (PRD 기능 목록 기준) | `F001` |
| 화면명 | 필수 | 기능이 위치하는 화면/페이지명 | `프로젝트 목록` |
| 트리거 | 필수 | 기능이 실행되는 조건/이벤트 | `페이지 진입`, `버튼 클릭`, `입력값 변경` |
| 입력값 | 필수 | 사용자 입력 또는 시스템 파라미터. 없으면 `—` | `검색어(string)`, `페이지 번호(int)` |
| 처리 | 필수 | 시스템 내부 처리 로직 (API 호출, 상태 변경 등) | `GET /api/projects?limit=10&sort=latest` |
| 출력 | 필수 | 처리 결과로 화면에 나타나는 UI 변화 | `카드 그리드 렌더링 (10개)` |
| 엣지케이스 | 필수 | 예외 상황별 처리 (빈 상태/에러/권한 없음/로딩) | `데이터 없음: 빈 상태 메시지 + CTA` |
| 비즈니스 규칙 | 선택 | 조직 정책·제한이 있는 경우만 추가 | `무료 플랜: 프로젝트 3개 한도` |

---

## 2. 문서 헤더

각 기능명세서 문서 최상단에 포함:

```markdown
---
client: {클라이언트ID}
project: {클라이언트명}_{프로젝트명}
type: functional-spec
created: {YYYY-MM-DD}
version: v1.0
prd_version: {참조 PRD 버전}
approved: false
---
```

| 항목 | 내용 |
|------|------|
| 서비스명 | {서비스명} |
| 기준 PRD | {PRD 파일명 및 버전} |
| 브레이크포인트 | Mobile ≤767px / PC ≥768px |
| 작성일 | {YYYY-MM-DD} |
| 버전 | v1.0 |

## 3. 섹션 구조

각 화면(Page/Screen) 단위로 아래 4개 섹션을 반복 작성:

### 3.1. 표출 정보 (Displayed Elements)

화면에 시각적으로 표시되는 UI 요소 목록. 번호 + 요소명 + 설명.

```markdown
| # | 요소명 | 설명 |
|---|--------|------|
| 1 | 헤더 | 서비스 로고 + 네비게이션 메뉴 |
| 2 | 프로젝트 카드 | 프로젝트명, 생성일, 상태 뱃지 표시 |
| 3 | 더보기 버튼 | 추가 프로젝트 로드 (10개씩) |
```

### 3.2. 인터랙션 명세 (Interaction Spec)

7컬럼 기능명세 표. 하나의 화면에서 발생하는 모든 사용자 행동을 행별로 기술.

### 3.3. 상태 정의 (State Definition)

화면의 4가지 상태를 명시:

```markdown
| 상태 | 조건 | UI 처리 |
|------|------|---------|
| 정상 | 데이터 로드 성공 | 프로젝트 카드 목록 표시 |
| 로딩 중 | API 응답 대기 | 스켈레톤 UI (카드 형태 회색 블록) |
| 빈 상태 | 프로젝트 0개 | "아직 프로젝트가 없습니다" + [새 프로젝트 만들기] |
| 에러 | API 응답 실패 | "불러오기 실패" 메시지 + [다시 시도] 버튼 |
```

### 3.4. 브레이크포인트 분기 (Breakpoint)

기준: Mobile ≤767px / PC ≥768px

```markdown
| 구분 | Mobile (≤767px) | PC (≥768px) |
|------|-----------------|-------------|
| 레이아웃 | 1열 카드 리스트 | 3열 카드 그리드 |
| 네비게이션 | 하단 탭바 | 좌측 사이드바 |
| 더보기 버튼 | 화면 전체 너비 | 우측 정렬, 고정 너비 |
```

Mobile/PC 동작이 동일한 경우: "Mobile/PC 공통" 으로 명시.

## 4. 엣지케이스 작성 기준

모든 기능에 아래 4종 엣지케이스를 검토하여 해당 항목을 기술:

### 4.1. 빈 상태 (Empty State)

- 표시할 데이터가 없는 경우
- 반드시 포함: 안내 메시지 + CTA 버튼 유무
- 예: "아직 프로젝트가 없습니다 — [새 프로젝트 만들기] 버튼 노출"

### 4.2. 네트워크 에러 (Error State)

- API 응답 실패 / 타임아웃 발생
- 재시도 UI 여부, 에러 메시지 표시 방식 명시
- 예: "데이터 로드 실패 — '다시 시도' 버튼 노출, 이전 캐시 데이터는 유지"

### 4.3. 권한 없음 (Unauthorized)

- 비로그인 상태, 또는 해당 기능 접근 권한 없는 사용자
- 리다이렉트 경로 명시 필수
- 예: "비로그인 → /login 리다이렉트, 로그인 후 원래 경로 복귀"

### 4.4. 로딩 중 (Loading State)

- API 응답 대기 시 UI 처리 방식 구분:
  - 스켈레톤 UI: 콘텐츠 레이아웃 유지하며 회색 플레이스홀더
  - 스피너: 전체 화면 또는 특정 영역 중앙
  - 프로그레스바: 단계별 진행 표시
  - 로딩 아이콘 회전: 360도 무한 회전 (회전→멈춤→회전→멈춤 등 패턴 명시)

## 5. 상태별 동작 분리 기준

기능이 사용자 상태에 따라 다르게 동작하는 경우 별도 행으로 분리:

| 기능ID | 화면명 | 트리거 | 입력값 | 처리 | 출력 | 엣지케이스 |
|--------|--------|--------|--------|------|------|------------|
| F005-A | 프로젝트 상세 | 페이지 진입 (로그인) | 프로젝트 ID | 프로젝트 상세 데이터 조회 | 상세 정보 + 편집 버튼 노출 | 존재하지 않는 ID: 404 페이지 |
| F005-B | 프로젝트 상세 | 페이지 진입 (비로그인) | 프로젝트 ID | 접근 권한 확인 | /login 리다이렉트 | — |

분리 기준:
- 로그인 / 비로그인
- 권한 있음 / 없음 (관리자 / 일반 사용자 등)
- 데이터 있음 / 없음

---

## 6. 기능 ID 부여 규칙

- 형식: `F` + 3자리 숫자 (F001, F002, ..., F099, F100)
- 화면 단위로 그룹화: 홈 화면 F001~F005, 목록 화면 F006~F012 등
- 상태 분기: F005-A, F005-B 형식
- PRD의 기능 요구사항 순서와 일치시킴 (Must-Have 먼저, Should-Have 이후)
- 삭제된 기능 ID는 재사용하지 않음 (이력 추적용)
