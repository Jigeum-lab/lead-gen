# 컨텍스트 엔지니어링 가이드

지금랩 Product Agent 파이프라인에서 에이전트 호출 시 컨텍스트를 구성하는 표준 패턴.
**모든 스킬과 에이전트는 이 가이드를 따른다.**

---

## 1. 계층적 컨텍스트 모델 (Layered Context Model)

컨텍스트는 4개 레이어로 분리된다. 레이어가 높을수록 구체적이고, 낮을수록 범용적이다.

### Layer 0 — 글로벌 (Global)

**포함 대상**: 디자인 토큰, 작성 규칙, 시스템 수준 제약
**파일 위치**: `.claude/skills/design-system/SKILL.md` (시각 토큰), 루트·앱별 `CLAUDE.md`/`AGENTS.md` (작성·아키텍처 규칙)
**언제 포함**: 시각적 산출물(wireframe, figma) 또는 문서 품질 검사 필요 시

예시 내용:
- 컬러 토큰: Primary `#2563EB` (blue-600), Surface `#FFFFFF`
- 타이포그래피: Pretendard / Inter
- 간격 기준: 8px grid
- 개조식 문체 강제, 빈 수식어 금지

### Layer 1 — 프로젝트 (Project)

**포함 대상**: `product/{project}/product-context.md` 전체 (11개 필드)
**언제 포함**: 항상 포함. 모든 에이전트 호출의 기본 컨텍스트
**크기 제한**: product-context.md는 압축하지 않는다 — 항상 전체 포함

11개 필드:
1. 제품명
2. 한 줄 설명
3. 타겟 유저
4. 핵심 문제
5. 솔루션 가설
6. 기술 스택 / 제약
7. MVP 범위
8. 비목표
9. 성공 지표
10. 예산 / 일정
11. 클라이언트 연락처

### Layer 2 — 단계 입력 (Stage Inputs)

**포함 대상**: 현재 단계 바로 이전 단계의 산출물
**언제 포함**: 해당 산출물이 존재하는 경우. 크기에 따라 압축 가능
**주의**: 관련 없는 이전 단계 산출물은 주입하지 않는다

### Layer 3 — 태스크 (Task)

**포함 대상**: 현재 에이전트에게 주는 구체적 지시
**언제 포함**: 항상 포함. 가장 구체적이고 가장 마지막에 위치
**형식**: 출력 경로, 완료 기준, 금지 사항, 마커 규칙 포함

---

## 2. 단계별 레이어 포함 기준

| 단계 | L0 (디자인·스타일) | L1 (product-context) | L2 (이전 단계 산출물) | L3 (태스크) |
|---|---|---|---|---|
| research | - | 전체 | - | 리서치 지시 |
| PRD | - | 전체 | research.md (압축 가능) | PRD 작성 지시 |
| user-flow | - | 전체 | PRD.md (압축 가능) | IA·플로우 지시 |
| functional-spec | - | 전체 | PRD.md + ia-userflow.md | 기능명세 지시 |
| wireframe | design-system | 전체 | PRD.md + ia-userflow.md + functional-spec.md | 화면설계 지시 |
| figma | design-system (우선순위 높음) + brand.md | 전체 | wireframes.md | Figma 생성 지시 |
| qa | - | 전체 | PRD.md + wireframes.md | 검증 지시 |

**figma 단계는 L0를 두 겹 참조**: design-system(기본값) → brand.md(오버라이드).
brand.md 있으면 design-system의 컬러·타이포 토큰은 brand.md가 덮는다.

## 3. 컨텍스트 압축 규칙 (Context Compression)

### 압축 트리거

| 파일 크기 | 처리 방법 |
|---|---|
| <2,000자 | 전체 포함 — 압축 없음 |
| 2,000~5,000자 | 전체 포함 — 경고 없이 |
| 5,000~15,000자 | Haiku 모델로 핵심 30% 압축 후 포함 |
| >15,000자 | 섹션별 요약 + "전체 내용은 {파일명} 참조" 명시 |

### Haiku 압축 프롬프트 템플릿

```
다음 문서의 핵심만 {목표 글자수}자 이내로 요약하세요.
아래 항목은 반드시 원문 그대로 포함하세요:
- 수치 (금액, 기간, 비율, 사용자 수)
- 기능 ID (F001~Fn)
- 완료 기준 문장
- 하드 제약 조건 (기술 스택, 법적 제약)
- 경쟁사명, 제품명, 고유명사

요약 대상:
---
{문서 내용}
```

### 절대 압축 금지 항목

이 항목들은 크기와 무관하게 항상 원문 포함:
- 기능 ID (F001, F002 ...)
- 완료 기준 문장
- 하드 제약 조건
- 수치 데이터 (숫자, 퍼센트, 금액)
- `[확인 필요]` / `[결정 필요]` 마커가 붙은 항목

## 4. 컨텍스트 주입 템플릿 (Injection Template)

모든 에이전트 호출 시 이 형식을 사용한다. 레이어 구분선으로 경계를 명확히 한다.

### 표준 템플릿

```
=== [LAYER 0: GLOBAL] ===
{design-system 토큰 요약 — 해당 단계만}

=== [LAYER 1: PROJECT CONTEXT] ===
{product-context.md 전체}

=== [LAYER 2: STAGE INPUTS] ===
[파일명: research.md | 버전: v1.2 | 원본 크기: 8,200자 → 압축본]
{압축된 내용 또는 전체}

=== [LAYER 3: CURRENT TASK] ===
작업: {구체적 지시}
출력 경로: product/{project}/{파일명}
완료 기준: {검증 가능한 조건}
금지 사항: {하지 말아야 할 것}
마커 규칙: 미확인 사항 → [확인 필요: {내용}], 미결 사항 → [결정 필요: {내용}]
```

### 레이어 생략 규칙

- Layer 0 없음: research, PRD, user-flow, functional-spec, qa 단계
- Layer 2 없음: research 단계 (최초 단계이므로 이전 산출물 없음)
- 레이어가 비어 있으면 해당 헤더 자체를 생략 (빈 섹션 금지)

---

## 5. 델타 컨텍스트 (Delta Context — 반복 작업용)

`/iterate` 스킬 또는 피드백 기반 수정 시 사용. 전체 컨텍스트 재주입 대신 변경 분만 주입.

### 델타 컨텍스트 구조

```
=== [DELTA CONTEXT] ===
현재 버전 요약: {이전 산출물의 핵심 — 기능 목록, 섹션명, 완료 기준}
변경 요청: {피드백 원문 또는 요약}
변경 대상: {섹션명 또는 기능 ID — "F003 로그인 기능", "§4.2 비기능 요구사항"}
불변 요소: {변경하지 말 것 — 피드백에서 언급 안 된 모든 것}
변경 이유: {왜 이 변경인지 — 사용자 의도 또는 DA 비평 결과}

=== [LAYER 1: PROJECT CONTEXT] ===
{product-context.md — 연관 필드만 발췌}
```

### 불변 요소 도출 방법

피드백에서 명시적으로 언급된 섹션·기능을 변경 대상으로 분류하고,
나머지 전체를 불변 요소로 자동 지정한다.

예시:
- 피드백: "3.2 로그인 기능의 엣지케이스가 부족해"
- 변경 대상: §3.2 로그인 기능 — 엣지케이스 섹션
- 불변 요소: §3.2 이외 모든 섹션, 기능 ID 체계, 완료 기준

## 6. 컨텍스트 오염 방지 (Context Pollution Prevention)

### 하지 말아야 할 것

**대화 히스토리 통째 주입 금지**
이전 대화의 검토 의견, 중간 초안, 취소된 결정은 컨텍스트에 포함하지 않는다.
에이전트는 현재 파일 상태와 명시적 지시만으로 동작해야 한다.

**무관한 단계 산출물 주입 금지**
wireframe 생성 시 research.md를 주입하는 것은 노이즈다.
wireframe 단계의 Layer 2는 ia-userflow.md + functional-spec.md만.

**정밀 데이터 압축 금지**
기능 ID, 완료 기준, 제약 조건은 압축 과정에서 손실될 수 있으므로 원문 유지.
Haiku 압축 프롬프트에 "반드시 포함" 목록을 명시하는 이유.

**중복 레이어 주입 금지**
같은 내용이 Layer 1과 Layer 2에 동시에 들어가는 것은 토큰 낭비.
product-context.md는 Layer 1에만. research.md는 Layer 2에만.

**빈 레이어 헤더 금지**
해당 레이어가 없으면 헤더(`=== [LAYER N: ...] ===`)도 생략.
빈 섹션은 에이전트 혼란을 유발한다.

### 오염 감지 체크리스트

에이전트 호출 전 컨텍스트 검토:
- [ ] 각 레이어에 해당 단계에 필요한 정보만 들어있는가?
- [ ] 압축된 문서에 기능 ID·수치·완료 기준이 살아있는가?
- [ ] 불변 요소가 명확히 구분되었는가? (Delta Context 사용 시)
- [ ] 이전 대화의 취소된 결정이 섞여 있지 않은가?
- [ ] Layer 0(글로벌)이 필요 없는 단계에 포함되지 않았는가?
